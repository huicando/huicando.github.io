###Linux 下安装MySql

#####安装：

rpm -ivh rpm 软件名
如果安装时与某个软件 XXX冲突，则需要将冲突的软件卸载掉：

yun -y -remove xxx

安装时 有日志会提示我们可以修改密码：输入

‘new-password’

注意：如果提示“GPG keys…”安装失败，解决办法：

rpm -ivh rpm 软件名 –force –nodoeps

验证：

mysqladmin –version

启动：

service mysql strat

关闭：

service mysql stop

重启：

service mysql restart

#####登陆：
登陆mysql（需要MySQL已经启动。）:mysql

MySQL 未启动的解决办法。
1.手动启动。service mysql strat
2.设置开启自动启动：chkconfig mysql on/off 检查是否开机自启：ntsysv

给root添加密码：/user/bin/mysqladmin -u root password root
登陆：mysql -u -root -p
MySQL字符编码: show variables like 'char'; 需要统一修改为utf-8
需要设置编码：

vi /etc/my.cnf:
[mysql]

default-character-set=utf8

[client]

default-character-set=uft8

[mysqld]

character_set_server=utf8
character_set_client=uft8
collation_server=utf8_general_ci

重启后生效。
注意：修改编码只对之后创建数据库生效，建议在安装完MySQL之后，第一时间设置

#####原理：
Mysql分层：

1.连接层：提供与客户端连接的服务
2.服务层：使用各个用户使用的接口；提供SQL优化器（MySQL QUery Optimizer）
3.引擎层：提供了各种存储数据的方式（InnoDB(事务优先：适合高并发操作；行锁) ;MyISAM(性能优先：表锁)）
4.存储层：存储数据

查看数据库引擎：
支持哪些引擎：show engines;
查看当前引擎：show variables like '%storage_engine%';
指定数据库对象的引擎: 

create table tb(
    id int(4) auto_increment,
    name varchar(5),
    dept varchar(5),
    primary key(id)
)ENGINE=MyISAM AUTO_INCREMENT=1
DEFAULT CHARSET=utf8;

#####SQL优化：
原因：

性能低，执行时间太长，SQL语句欠佳（连接查询），索引失效，服务器参数设置不合理（缓冲，线程数）

1.sql语句的编写和解析
2.SQL优化，主要是 在优化索引。

索引：相当于书的目录。
索引：index是帮助MySQL高效获取数据的数据结构（树，hash结构）。
索引的弊端：

1.本身很大，可以存放在内存/硬盘。
2.不是所有情况均可适用: a.少量数据 b.频繁更新的字段 c.很少使用的字段
3.索引会降低增删改的效率。

索引的优势：

1.提高查询的效率(降低IO使用效率)。
2.降低cpu的使用

索引的分类：

单值索引：单列，一个表可以多个单值索引。
唯一索引：单列，不能重复，一般是主键
复合索引：多列构成的索引。（相当于二级目录）

创建索引：

方法一：create 索引类型 索引名 on 表（字段）
单值：create index dept_index on tb(dept);
唯一：create unique index name_index on tb(name);
复合：create index dept_name_index on tb(dept,name);
方法二：alter table 表名 索引类型 索引名（字段）
单值：alter table tb add index dept_index(dept);
唯一:alter table tb add unique index name_index(name);
复合：alter table tb add index dept_name_index(dept,name);
注意：如果一个字段是主键，则该字段默认就是 主键索引

删除索引：drop index 索引名 on 表名;

drop index name_index on tb;

查询索引：show index from 表名；

show index from tb;

SQL性能问题：
1.分析SQL的执行计划。explain，可以模拟SQL优化器执行SQL语句，从而让开发人员编写执行语句。
2.MySQL查询优化器会干扰我们的优化。
查询执行计划：explain+SQL语句

explain select * from tb;

#####explain中的列解析：
(1)id：

id值相同，从上到下；id值不相同，越大越优先

(2)select_type:（查询类型）

PRIMARY:包含子查询sql的主查询（最外层）
SUBQUERY:包含子查询sql中的子查询（非最外层）
simple：简单查询（不包含子查询，union）
derived：衍生查询（使用到临时表）

1.在from子查询中只有一张表
2.在from子查询中，如果有table1 union table2，则table1 就是 derived，table2 就是union

union:连接
union result：告知开发人员，哪些表存在union查询。

(3)type:索引类型，类型
system>const>eq_ref>ref>range>index>all

越左边性能越高(要对type进行优化的前提：有索引)
其中system,const只是理想情况，实际能达到ref>range
system:只用一条数据的系统表，或者衍生表只用一条数据的主查询
const：仅仅能查到一条数据的SQL,用于Primary key或unique索引（类型与索引类型有关）
eq_ref:唯一性索引：对于每个索引键的查询，返回匹配唯一行数据（有且只有1个，不能多，不能0）（查询结果行数与表数据相同且要一一对应）
ref:非唯一性索引，对于每个索引键的查询，返回匹配的所有行（0对多）
range：检索指定范围的行，where后面是一个范围查询（betwee，>，<，in(有时候会失效，转化为all)）
index：查询全部索引中的数据。
all:查询全部表中的数据。

(4)possible_keys:可能用到的索引(预测)。
(5)key:实际使用到的索引。

如果possible_keys,keys都为null,则说明没有索引。

(6)key_len:索引的长度；
作用：用于判断复合索引是否被完全使用

如果索引可以为nll,则会用一个字节用于标识。

(7)ref:注意与type中的ref区分。
指明当前表所参照的字段。
(8)rows:
被索引优化查询的 数据个数（实际通过索引而查询到的数据个数）
(9)Extra:(额外的)

using filesort:性能消耗大，需要”额外”的一次排序（常出现在order by）

对于单索引，如果排序和查找的是同一个字段，则不会出现，如果不是相同的就会出现
复合索引:如果跨列排序使用就会出现，where 和order by 按照复合索引的顺序使用，不要跨列或无序使用。

using temporary:性能损耗大，用到了临时表。(一般出现在group by)

避免：查询那些列，就根据那些列group by

using index:性能提升，索引覆盖。

原因：不读取原文件，只从索引中获取数值。只要使用到的列全部都在索引中，就是索引覆盖using index。
如果用到了索引覆盖（using index），会对possible_keys 和key造成影响:

如果没有where，则索引只出现在key中
如果有where，则索引出在key和possible_keys中。

using where(需要回表查询)

####优化示例

#####单表优化：
加索引：

最佳左前缀，保持索引的定义和使用的顺序的一致性。
索引需要逐步优化。
将含IN的范围查询放到where的最后面，防止失效。

#####两表优化：
加索引：

小表驱动大表。
索引建立在常用的表的字段上。(左连接建立在左表上)(右连接建立在右表上)

#####多表优化：

与双表优化基本相同。
小表驱动大表
索引建立在经常查询的字段上

####避免索引失效的一些原则：
(1)复合索引：

不用跨列或者无序使用(最佳左前缀)
尽量使用全索引匹配，把该有的都用上

(2)不要再索引上进行任何操作（计算、函数、类型转换),否则索引失效。
(3)不能使用不等于(!= ,<>)或 is null(is not null),否则自身以及右侧所有全部失效。

一般而言(> < in)范围查询，之后的索引查询，因为优化器等原因，结论不是100%正确

(4)补救。尽量使用索引覆盖（using index）
(5)like尽量以“常量”开头，不要以”%”开头，否则用索引失效。如果必须使用”%”尽量使用索引覆盖。
(6)尽量不要使用or,否则索引失效。

#####其他优化方法：
(1)exist 和 in

in:适用于主查询的数据集大。
exist：使用于子查询数据集大。

exist:将主程序的结果，放到子查询结果中进行条件效验（是否有数据，如果有数据，校验成功，如果符合校验，则保留数据）

(2)order by 优化：
using filesort:有两种算法：双路排序，单路排序。

双路排序:扫描两次磁盘（1：从磁盘读取排序字段,对排序字段进行排序 2.扫描其他字段）
单路排序：只读取一次(全部字段)，在buffer中进行排序(如果数据量太多，会拆分多次排序)
单路排序会比双路排序占用更多的buffer
set max_length_for_sort_data设置buffer的最大字节。

提高order by 查询的策略：

避免使用单路，双路；调整buffer的容量大小；
避免select *。。。尽量索引覆盖。
复合索引不用跨列使用，避免using filesort
保证全部的排序字段 排序的一致性（都是升序或降序）

####SQL排查

慢查询日志：MySQL提供的一种日志记录，用于记录mysql中响应时间超过阀值的sql语句。(建议：开发调忧时打开，而最终部署时关闭)。

检查是否开启了慢查询日志：show variables like '%slow_query_log%';
临时开启：set global slow_query_log=1；重启mysql后会失效。
慢查询阀值：show variables value like'%long_query_time';
临时设置阀值：set global long_query_time=5;需要重新登陆。
永久设置：在配置中添加long_query_time=3
查询超过阀值的sql:show global status like '%show_queries%';

(1)慢查询的sql被记录在日志中，因此可以通过日志查看具体的慢sql。
（2）通过mysqldumpslow 工具查看慢sql,可以通过一些过滤条件快速查找出需要定位的慢sql:mysqldumpslow --help

s:排序方式
r:逆序
l:锁定时间
g:正则匹配模式

语法:mysqldumpslow 各种参数 慢查询日志文件

查看状态:show variables like '%profiling%'

####锁机制：
解决因资源共享，而造成的并发问题。
分类：

  • 操作类型：

    读锁(共享锁)：对同一数据，多个读操作可以同时进行，互不干扰
    写锁(互斥锁)：如果当前写操作没有完毕，则无法进行其他的读操作、写操作。

  • 操作范围：

    表锁:一次性对一张表整体加锁。(开销小，加锁快；无死锁；但锁的范围大，容易发生锁冲突，并发度低)。
    行锁：y一次性对一条数据加锁。(开销大，加锁慢，容易出现死锁，锁的范围小，不容易出现锁冲突，并发度高)
    页锁：

查看加锁的表：show open tables;
(1)加读锁：
加读锁：lock table tablelock read
释放锁：unlock tables；
会话:session :代表每个访问数据库的的用户。

会话0：对某个表加读锁。

如果只给A表加读锁，则当前会话只能对A表进行读操作。

会话1(其他会话)：

对于表A可读，对表A的插入删除操作，一直再等待，等到表被放开。
对其他表也可读可写。

会话0给A表加锁；其他会话的操作，可以对其他表进行操作，对A表，可读，写需要等到A表被释放锁。

(2)加写锁：
加写锁：lock table tablelock write;

会话0：

当前会话，可以对加了写锁的表，进行任何操作；但是不能对其他表进行操作。

其他会话：

对会话0加锁的表，可以进行操作，但是要等到会话0释放写锁。

分析表锁定：

查看加锁的表：show open tables;
分析表锁定的严重程度：show status like 'table%';

Table_locks_immediate:即可能获取到的锁数。
Table_locks_waited:需要等待的表锁数(如果该值越大，说明存在的锁竞争越大)

一般建议：Table_locks_immediate/Table_locks_waited>5000,建议采用InnDB引擎。

(2)行锁:

需要暂时将自动commit关闭，set autocommit=0;以后需要通过commit
对行锁情况：

1.如果会话X对某条数据a进行DML操作（研究时：关闭了自动commit的情况下），其他会话必须等待会话X结束事务后，才对数据a进行操作
2.表锁是通过unlock tablesl行锁是通过事务解锁。

注意事项：
如果没有是索引，则行锁会转化为表锁。
行锁的一种特殊情况：间隙锁:值在范围内，但却不存在。

间隙锁：Mysql会自动给 间隙 加锁–>间隙锁。
行锁：如果有where，则实际加锁范围就是where后面的范围（不是实际范围）

行锁：
InnoDB默认采用行锁。

缺点：比表锁性能损耗大。
优点：并发能力强，效率高。

高并发用InnoDB,否则用MyISAM

行锁分析：
show status like '%innodb_row_lock%';

Innodb_row_lock_current_waits:当前正在等待锁的数量。
Innodb_row_lock_time:等待总时长。从系统开启到现在，一共等待的时间。
Innodb_row_lock_time_avg:平均等待时长。从系统开启到现在平均等待的时间。
Innodb_row_lock_time_max：最大等待时长，从系统启动到现在最大一次等待的时间。
Innodb_row_lock_waits:等待次数，从系统开启到现在一个等待的次数。

主从复制：
远程连接linux:数据库。

GRANT ALL PRIVILEGES ON *.* TO 'root' @'%' IDENTFIED BY 'root' WITH GRANT OPTION;
FLUSH PRIVILEGES;
如果仍然报错可能是防火墙没关闭：在 远程的上关闭防火墙

原理：

1.master 将改变的数记录在本地的二进制日志中（binary log）;该过程称之为二进制日志事件。
2.slave 将master的binary log 拷贝到自己的relayl log （中继日志文件）中
中继日志文件，将数据读取到自己的数据库之中

Mysql主从复制是异步的，串行化的，有延迟。
